# Phase 2: VOC Intelligence Demo — 설계

## 전체 아키텍처

```
BigQuery 원본 (편지글/게시글/채널톡)
    ↓
분류 파이프라인 (KcELECTRA + LLM fallback)
    ↓
┌──────────────────────────────────────┐
│        분류 완료 데이터 저장소          │
│  ├─ topic + sentiment + detail_tags  │  ← 편지글/게시글
│  └─ route + topics[]                 │  ← 채널톡
└──────────────┬───────────────────────┘
               ↓
    ┌──────────┼──────────┬───────────┬──────────┐
    ↓          ↓          ↓           ↓          ↓
 대시보드    리포트v2   AI Agent   Slack Bot   자동 알림
(Streamlit)  (마크다운)  (자연어)   (슬래시)    (푸시)
```

## 1. 핵심 프로토타입

### 1-A. 부서별 대시보드 (Streamlit)

기존 `dashboard/app.py` 확장. 3개 데이터 소스를 통합한 부서별 뷰.

#### 탭 구조

```
[전사 개요] [운영/사업팀] [콘텐츠팀] [서비스/개발팀] [원문 탐색]
```

#### 전사 개요 탭
- 이번 주 전체 통계 (편지/게시글/채널톡 건수, 전주 대비)
- Topic × Sentiment 히트맵
- 마스터별 부정 비율 랭킹 (위험신호)
- 주차별 감성 추이 차트

#### 운영/사업팀 탭
| 기능 | 데이터 소스 | 설명 |
|------|-----------|------|
| 마스터 헬스 스코어 | 편지글+게시글 | 마스터별 긍/부정 비율 추이, 전주 대비 변화 |
| 위험신호 감지 | 편지글+게시글 | 부정 비율 급증 마스터 자동 하이라이트 |
| 운영 피드백 큐 | 편지글+게시글 | topic=운영 피드백, 카테고리별 건수 + 대표 사례 |
| 상품 런칭 반응 | 편지글+게시글 | 특정 기간 + 자유태그 검색으로 상품 반응 추적 |
| 투자 여론 트렌드 | 편지글+게시글 | topic=투자 담론, 주차별 감성 추이 (증시 연동) |

#### 콘텐츠팀 탭
| 기능 | 데이터 소스 | 설명 |
|------|-----------|------|
| 콘텐츠 반응 분석 | 편지글+게시글 | topic=콘텐츠 반응, 감성별 카테고리태그 분포 |
| 부정 패턴 분석 | 편지글+게시글 | 콘텐츠 부정 의견의 카테고리/자유태그 집계 |
| 마스터별 콘텐츠 피드백 | 편지글+게시글 | 마스터별 콘텐츠 관련 긍/부정 요약 |
| 주제 요청 트렌드 | 편지글+게시글 | category_tag=콘텐츠 주제 요청 추이 |

#### 서비스/개발팀 탭
| 기능 | 데이터 소스 | 설명 |
|------|-----------|------|
| 서비스 이슈 통합 뷰 | 편지글+게시글+채널톡 | **3개 소스 교차 분석** |
| 채널톡 이슈 분류 | 채널톡 | route별 건수 + topic별 분포 |
| 반복 이슈 패턴 | 채널톡+편지글 | 같은 이슈가 양쪽에서 발생하는지 |
| 기술 이슈 큐 | 편지글+게시글 | topic=서비스 피드백, category_tag별 건수 |
| 해결률 추적 | 채널톡 | route=manager_resolved vs bot_resolved 비율 추이 |

#### 원문 탐색 탭
- 전체 분류 데이터 테이블 (필터링 가능)
- 조건: 마스터, 기간, topic, sentiment, category_tag, 자유태그, 키워드
- 원문 클릭 시 전체 텍스트 + 분류 메타데이터 표시

### 1-B. 리포트 생성기 v2 (파이프라인 기반)

기존 리포트(`example.md` 형식) 대체. 핵심 차이: LLM이 원문을 전수 읽지 않음.

#### 기존 vs 신규 비교

| | 기존 | 리포트 v2 |
|---|---|---|
| **입력** | 원본 텍스트 2-3K건 | 분류 완료 데이터 (topic/sentiment/tags/summary) |
| **LLM 역할** | 분류 + 그룹핑 + 요약 + 서술 | 요약 + 서술만 |
| **비용** | 높음 (전수 읽기) | 낮음 (클러스터 대표 원문만) |
| **속도** | 느림 (수십 분) | 빠름 (수 분) |
| **일관성** | LLM 판단에 의존 | 분류 모델 기준 일관 |

#### 생성 파이프라인

```
1. 분류 데이터 로드 (이번 주 + 전주)
   ↓
2. 전처리
   - 마스터별 그룹핑 (masterName)
   - 각 마스터 내 topic × sentiment 집계
   - category_tags 빈도 집계 → 주요 주제 클러스터 도출
   - 전주 대비 증감 계산
   ↓
3. 클러스터별 대표 원문 선정
   - 각 category_tag 클러스터에서 상위 3-5건 (summary 기준)
   - 서비스 피드백은 전수 포함
   ↓
4. LLM 요약 (마스터별)
   - 입력: 통계 + 클러스터 + 대표 원문 (건별 summary + 원문)
   - 출력: 마스터별 섹션 (주요 내용, 서비스 피드백, 체크포인트)
   ↓
5. 리포트 조립
   - 핵심 요약 (전체 통계 테이블)
   - 마스터별 상세 (example.md 형식)
   - 부록: 카테고리 분류 기준 안내
```

#### 출력 형식
- 기존 `example.md`와 동일한 마크다운 형식 유지
- 핵심 요약 → 마스터별 상세 → 서비스 피드백 → 체크포인트
- 직접 인용은 이탤릭 (_"인용문"_)
- 개선 권고는 화살표 (_→ 권고사항_)

#### 부서별 맞춤 리포트 (옵션)

동일 분류 데이터에서 다른 필터로 부서별 리포트 생성:

| 리포트 | 필터 | 초점 |
|--------|------|------|
| 전사 리포트 | 전체 | 기존 example.md와 동일 |
| 운영 리포트 | topic=운영 피드백 + 부정 전체 | 대응 필요 건만 |
| 콘텐츠 리포트 | topic=콘텐츠 반응 | 콘텐츠 피드백만 |
| 서비스 리포트 | topic=서비스 피드백 + 채널톡 | 기술 이슈 통합 |

### 1-C. 채널톡 × 편지글 교차 분석

같은 이슈가 CS(채널톡)와 커뮤니티(편지글)에서 동시에 터지는 패턴 감지.

#### 교차 분석 로직

```
채널톡 topics                    편지글/게시글 topic + category_tags
─────────────                    ─────────────────────────────────
결제·환불          ←→            서비스 피드백 + 결제/환불 정책
구독·멤버십        ←→            운영 피드백 + 구독/멤버십 관리
콘텐츠·수강        ←→            콘텐츠 반응 + 콘텐츠 접근 문제
기술·오류          ←→            서비스 피드백 + 앱/기능 오류
```

#### 인사이트 예시
```
⚠️ "결제 이슈" 교차 감지 (2026-02-10 주차)
  - 채널톡: 결제·환불 12건 (전주 대비 +8건)
  - 편지글: 서비스 피드백 + 결제 관련 5건 (전주 대비 +3건)
  → 동시 급증 → iOS 결제 시스템 장애 가능성 높음
```

## 2. 확장 레이어 (구조 제시)

### 2-A. AI Agent (자연어 질의)

> 기존 `analysis-scenarios.md`의 시나리오 1~7을 자연어로 수행

#### 구조

```
사용자 질문
  ↓
LLM: 질문 → 구조화된 쿼리로 변환
  {master: "미과장", topic: "콘텐츠 반응", sentiment: "부정", period: "this_week"}
  ↓
분류 데이터 필터링 (코드)
  ↓
LLM: 필터 결과 → 인사이트 요약
  ↓
응답 + 근거 데이터 건수 + 원문 보기 옵션
```

#### 지원 질문 유형

| 유형 | 예시 | 쿼리 변환 |
|------|------|----------|
| 부서별 조회 | "이번 주 서비스 이슈 뭐 있어?" | topic=서비스 피드백, period=this_week |
| 마스터별 조회 | "박두환 부정 여론 요약해줘" | master=박두환, sentiment=부정 |
| 트렌드 | "지난 4주 투자 감정 추이" | topic=투자 담론, period=4weeks, aggregate=weekly |
| 상품 반응 | "프리미엄 멤버십 반응 어때?" | free_tags CONTAINS 멤버십,프리미엄 |
| 교차 분석 | "결제 관련 CS랑 커뮤니티 반응 비교" | channel=결제·환불 + letters=서비스+결제 |
| 전사 요약 | "이번 주 핵심 3줄" | all, period=this_week, format=summary |

#### 검증 경로
- 모든 답변에 **근거 건수** 표시 ("서비스 피드백 12건 기반")
- 원문 보기 옵션 → 분류 정확도를 사용자가 직접 확인
- 오분류 피드백 → 모델 개선 사이클 연결

### 2-B. Slack Bot

AI Agent를 Slack에서 호출하는 인터페이스.

```
/voc 미과장 이번주 부정 여론
/voc 서비스이슈 요약
/voc 채널톡 결제 이슈 추이
```

- 내부적으로 2-A의 AI Agent와 동일한 파이프라인
- Slack webhook으로 결과 전송
- 스레드에서 후속 질문 가능

### 2-C. 자동 알림 (Proactive Monitoring)

분류 파이프라인 실행 시 자동으로 이상치를 감지하고 Slack 채널에 푸시.

#### 알림 규칙 예시

| 규칙 | 조건 | 대상 채널 |
|------|------|----------|
| 부정 급증 | 마스터별 부정 비율 전주 대비 +10%p 이상 | #voc-운영 |
| 서비스 장애 | 채널톡 특정 topic 건수 전주 대비 2배 이상 | #voc-서비스 |
| 결제 이슈 | 채널톡 결제·환불 + 편지 결제 교차 급증 | #voc-서비스, #voc-운영 |
| 콘텐츠 불만 | 콘텐츠 반응 부정 5건 이상 동일 마스터 | #voc-콘텐츠 |

#### 알림 형식
```
⚠️ [부정 급증] 박두환 마스터
이번 주 부정 비율: 28% (전주 15%, +13%p)

주요 원인:
- 투자 담론·부정 45건: "수익인증 요구 논란"
- 콘텐츠 반응·부정 8건: "분석 깊이 부족"

→ 자세히 보기: [대시보드 링크]
```

### 2-D. 주간 자동 발행

리포트 v2를 cron/스케줄러로 자동 생성 + 배포.

```
매주 월요일 09:00
  ↓
BigQuery에서 전주 데이터 추출
  ↓
분류 파이프라인 실행
  ↓
리포트 v2 생성
  ↓
Slack #voc-주간리포트 채널에 자동 게시
+ Notion/Google Docs에 자동 저장 (옵션)
```

## 3. 데이터 준비 사항

### 과거 데이터 2-axis 재분류

현재 2-axis 데이터가 2주치(2026-02)만 존재. 트렌드 분석 데모를 위해 과거 데이터 재분류 필요.

| 작업 | 설명 |
|------|------|
| 1-axis → 2-axis 마이그레이션 | 기존 11주치 데이터를 2-axis 파이프라인으로 재분류 |
| detail_tags 추가 | category_tags, free_tags, summary를 LLM으로 추출 |
| 채널톡 과거 데이터 | BigQuery에서 추가 추출 + 분류 파이프라인 적용 |

### 데이터 저장 구조

```
data/
├── classified_data_two_axis/     ← 편지글/게시글 (2-axis + detail_tags)
│   ├── 2025-12-08.json
│   ├── ...
│   └── 2026-03-16.json
├── channel_io/
│   ├── classified/               ← 채널톡 (route + topics[])
│   │   ├── 2025-11-24.json
│   │   └── ...
│   └── golden/
└── stats/                        ← 주간 집계 통계 (대시보드 캐시)
```

## 4. 기술 스택

| 컴포넌트 | 기술 | 비고 |

| 분류 모델 | KcELECTRA (로컬) | 기존 |
| LLM fallback | Claude Haiku | 기존 |

## 5. 성공 기준

| 기준 | 측정 방법 |
|------|----------|
| 리포트 품질 | example.md와 동등 이상의 인사이트 (사업팀 주관 평가) |
| 리포트 속도 | 기존 대비 5배 이상 빠름 |
| 리포트 비용 | 기존 대비 70% 이상 절감 |
| 대시보드 유용성 | 부서별 담당자가 "이거 쓸 수 있겠다" 반응 |
| 교차 분석 | 채널톡+편지글 교차 이슈 1건 이상 식별 |
| AI Agent | 시나리오 7개 중 5개 이상 자연어로 수행 가능 |
