# 피드백 벡터 클러스터링 → Jira 태스크 자동화

## 목적
편지글/게시글 피드백을 벡터 임베딩 기반으로 클러스터링하여, 개발팀이 Jira 태스크로 바로 대응할 수 있는 구조로 자동 생성한다.

## 현재 완료 (Phase 1)

### 파이프라인
1. BigQuery에서 편지/게시글 조회
2. v5 Bedrock Haiku로 4분류 (topic + subtag)
3. 피드백 중 이용문의 제외
4. LLM이 각 건에 대해 세부 증상 자유 서술 (고정 카테고리 없음)
5. 서술 + 원문을 Bedrock Titan으로 임베딩
6. Agglomerative Clustering (코사인 거리 + 키워드 겹침 보정)
7. 클러스터 = Jira 태스크 단위로 엑셀 출력 + 슬랙 전송

### 핵심 설계 결정
- **고정 카테고리 대신 벡터 클러스터링**: 기타/미분류 문제 해소, subtag 경계 넘는 이슈 자동 병합
- **임베딩 대상**: `LLM 서술 | 원문` concat → 서술 스타일 차이를 원문 유사도가 보완
- **키워드 보정**: tags 필드의 Jaccard 유사도로 거리 40% 감쇄 → 동의어 문제 완화
- **threshold 0.6**: 128건 기준 49개 클러스터, 2건+ 31개

### 관련 파일
- 클러스터링 스크립트: `scripts/cluster_tech_issues.py`
- 1차 분류 + 엑셀 스크립트: `scripts/generate_tech_issues_report.py`
- Streamlit 뷰어: `dashboard/tech_issues_viewer.py`
- 출력: `exports/tech_issues_clustered_*.xlsx`

## Phase 2: LLM 검수 (다음 계획)

### 문제
- 클러스터링 후 오분류 존재 ("2달째 리포트 안보는중" → 사용자가 안 본 건데 기술이슈로 분류)
- 원문이 모호한 경우 LLM 서술이 부정확 → 잘못된 클러스터 배치
- 같은 이슈인데 서술 차이로 다른 클러스터에 들어간 경우

### 해결: 클러스터 단위 LLM 검수
클러스터링 완료 후, 각 클러스터의 원문을 LLM에게 보여주고 검수:

```
입력: 클러스터 C21 (12건)
  대표 서술: "줌 라이브 접속 불가"
  원문 1: "줌에 들어가지지 않아요"
  원문 2: "라이브 수업에 들어가기가 안됩니다"
  ...

질문:
1. 이 클러스터에 안 맞는 건이 있는가?
2. 다른 클러스터에서 여기로 옮겨야 할 건이 있는가?
3. 기술/서비스 이슈가 아닌 건이 포함되어 있는가?
```

### 기대 효과
- 오분류 보정 (개인 사유 vs 기술이슈 구분)
- 클러스터 간 병합/분리 제안
- 최종 Jira 태스크 품질 향상

### 비용 추정
- 클러스터당 원문 3~5건 × 50개 클러스터 = ~200건 읽기
- Haiku 기준 $0.5 이하

## Phase 3: 주간 파이프라인 통합 (이후)

### 목표
매주 리포트 생성 시 자동으로 피드백 클러스터링 + Jira 태스크 생성

### 구현
1. `generate_weekly_report.py`에 클러스터링 단계 추가
2. Jira API 연동 → 클러스터별 자동 티켓 생성
3. 이전 주 클러스터와 비교 → 반복 이슈 감지, 신규 이슈 하이라이트
4. 슬랙 알림에 클러스터 요약 포함
